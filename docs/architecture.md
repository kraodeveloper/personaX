# personaX Agent 系统架构

本篇是 personaX 的 **agent 系统架构文档**(架构层,不是技术实现层)。规定整体分层、组件职责、关键机制、边界与开放项。收敛过程见 [business-agent-convergence.md](business-agent-convergence.md)。

---

## 1. 定位

personaX 是一个 **分层上下文边界 Agent 系统**,面向大厂服务端的多业务域工作场景。

一句话:

> 以 Global Lead 为任务入口、以**版本化的领域知识库**为长期上下文底座、以**注册管理的领域 Agent** 持有上下文边界、以 Worker 做局部执行;跨边界只传结构化结论与证据引用。

**价值不在 agent 数量,也不在调用编排本身。** 调用、路由、派发用现成 agent 能力即可(会被通用 coding agent 持续覆盖)。真正的产品本体是:

> **被调用的那个 agent 自带领域知识——而"领域知识如何被组织成可版本化、可治理、可注入的资产,以及如何按身份调用承载它的 agent",是自研核心。**

---

## 2. 核心红线

贯穿全部设计,任何版本不得违反:

1. **Agent 是被注册管理的一等公民,不是运行时凭空 spawn 的临时线程。** 持久的是 agent 定义与知识库;运行时只"调用 + 实例化",不创造不可管理的 agent。
2. **LLM 只能 propose / critique,知识库的状态转移由确定性治理 policy 决定。**
3. **知识库不原地修改。** 任何沉淀都先成 candidate / patch proposal,经验证或人工确认后生成新版本。
4. **跨 Agent 不传大块上下文,只传结构化结论与证据引用。**
5. **当前证据优先于 base。** base 是业务地图,不是事实最终来源;过期 base 不得被当作权威。

---

## 3. 整体分层

系统分三个面:**管理面(持久,人可管理)**、**运行编排面(任务驱动)**、**知识沉淀闭环**。

```
┌──────────────────────────────────────────────────────────────┐
│  管理面（持久层 · 你定义 / 版本化 / 治理）                       │
│                                                                │
│   Agent 注册表                     知识库（业务域 / 技术域）     │
│   - 预定义领域 agent               - 版本化 base                 │
│   - 绑定知识库 + 能力范围          - payment-kb@v12 ...          │
│   - 可上下线                       - 可治理                      │
└───────────────┬───────────────────────────┬───────────────────┘
                │ ① 调用 + 实例化            ▲ ④ 新版本回写
                │   + 注入对应知识库         │
┌───────────────▼───────────────────────────┴───────────────────┐
│  运行编排面（任务驱动 · 用现成 agent 能力）                      │
│                                                                │
│   用户任务 → Global Lead ──② 调用──> Business Domain Agent(s)    │
│                 │  路由/调度/汇总          │ (多业务域→多业务主)  │
│                 │                          ├─> Technical Domain  │
│                 │  <─③ 只回传 claim+证据──  │     Agent(s)        │
│                 ▼                          ▼                     │
│              最终交付                    Worker（一次性执行）     │
└────────────────────────────────────────────────────────────────┘
                                                  │
                            产出 → 沉淀 → 治理(patch·确认) → 新版本
```

编号对应运行时序:① 管理面提供 agent + 知识 → ② Lead 调用领域 agent → ③ 领域 agent 回传结构化结论 → ④ 产出经治理回写为知识库新版本。

## 4. 组件职责

### 4.1 管理面(自研核心)

**Agent 注册表**
- 持有每个领域 agent 的**定义**:身份、绑定的知识库、能力范围、可调用接口。
- 支持创建 / 版本化 / 上下线 / 治理。
- 是"调用已有 agent"的目录;Lead 从这里按域选 agent,而不是临时拼。

**知识库(版本化 base)**
- 每个业务域 / 技术域一个 base,版本化(`payment-base@v12`)。
- 内容是稳定地图:业务概念、核心链路、服务边界、状态机、字段语义、模块地图、关键配置含义、历史高频问题、团队确认约束。
- 不放临时脚印:当前日志、告警、一次性猜测、未验证结论。

### 4.2 运行编排面(用现成能力)

**Global Lead**
- 当前任务总入口:路由、拆解、调度、冲突仲裁、结论合成、最终交付。
- 不持有所有业务/技术细节,是协调器与领域路由器,不是超级专家。

**Business Domain Agent(s)——多业务域则多业务主 agent**
- 每个业务域(订单 / 支付 / 风控 / 履约 / 用户 / 营销)一个主 agent,持有本域 base。
- 负责本域内的语义判断(如"支付域是否认为该 payment_id 已业务成功")。
- 业务域本身就是 context boundary。

**Technical Domain Agent(s)**
- 持有技术域 base(日志 / 配置 / 发布 / CI / 监控 / 代码库)。
- 负责技术证据的获取与解释。
- 哪些证据获取需要独立技术 agent、哪些由业务主 agent 自行完成,是**粒度问题**(见 §7 开放项)。

**Worker**
- 短生命周期、无长期记忆,执行一次性工具调用 / 检索 / 命令。
- 直接复用现成 agent 的瞬态 fan-out 能力。

### 4.3 知识沉淀闭环(自研核心)

任务产出 → 提炼可沉淀信息 → 生成 base patch proposal → 验证 / 人工确认 → 创建新 base 版本。错误结论不污染长期底座,历史任务可追溯。

## 5. 关键机制

### 5.1 Agent 实例化模型:资产持久,实例瞬态

这是本架构的轴心:**你管理的是 agent 资产,不是一堆活着的进程。**

- **持久且被管理的是资产**:agent 定义 + 版本化知识库,常驻注册表 / 知识库。
- **运行时是资产的投影**:Lead 按任务从注册表选定 agent → 从其绑定 base **fork** 出一个任务实例 → 注入知识 → 执行 → 实例用完即弃。
- 任务实际上下文 = `Base Context + Delta Overlay + Run State`。
- 每个任务实例记录 `base_id / base_version / base_fingerprint / forked_at`,可追溯它基于哪份地图工作。

### 5.2 窄接口:跨 Agent 只传结论与证据

谁拥有某类长期上下文,谁就在本地展开它;跨 agent 只传结构化 claim,不复制原始日志 / 完整代码 / 大段上下文。

```json
{
  "claim": "payment-callback-service 在 14:03 收到渠道成功回调",
  "confidence": 0.93,
  "evidence_refs": ["log://trace_id=abc/span=callback/ts=14:03:21"],
  "relevant_excerpt": "短摘录",
  "open_questions": ["未确认 order-core 是否消费该事件"]
}
```

### 5.3 知识注入

调用领域 agent 时注入其绑定 base 的对应版本。**注入形式(静态整库 vs 按任务动态编译 capsule)本架构暂不锁定**,先保证"有知识被注入、可工作",形式作为开放项(见 §7)。

### 5.4 Base 版本化与过期

base 版本化、可 fork、可追溯、不原地改。过期检测与"过期但权威"的防护是已知难点(见 §7),架构上先确立原则:**当前证据优先于 base,base 仅作地图**,以降低过期 base 的危害。

## 6. 执行核与"原生 vs 自研"分界

运行编排面跑在 **Claude Agent SDK** 上,壳能用原生就用原生:

| 能力 | 归属 |
|---|---|
| agent 载体、调用、路由、Worker 瞬态 fan-out | 原生(SDK / 现成 agent) |
| 把静态、小、不变的知识灌进一个 agent | 原生(system prompt / skill) |
| 知识库版本化 + fork + 不原地改 | **自研** |
| 知识更新治理(candidate → 确认 → 新版本) | **自研** |
| 跨 agent 窄接口(claim / evidence / confidence) | **自研** |
| 按任务动态编译注入 capsule(若采用) | **自研** |

> 在一个活着的会话内,"带知识的 agent 可被再调用"是 SDK 原生能力;但"跨会话、按身份调用一个持久领域 agent"原生没有——这正是 §5.1 用"资产持久 + 实例 fork"补上的部分。

---

## 7. 开放项(待后续锁定)

1. **证据获取粒度**:哪些由业务主 agent 自行完成(如读代码),哪些需要独立 technical domain agent(如查日志)。
2. **知识注入形式**:静态整库 vs 动态编译 capsule;token 预算。
3. **业务域 base 划分依据**:按业务域 / 服务 / 代码库 / 链路。
4. **业务域 vs 技术域边界**的精确定义。
5. **Base 过期检测**与"过期但权威"防护。
6. **patch proposal 验证**:哪些更新可自动、哪些必须人工确认。
7. **多业务域结论冲突时 Global Lead 的仲裁机制**。
8. **Global Lead 的最小必需上下文**到底多小。
9. 相对单 agent / 普通 RAG / 普通 skill 编排的**真实收益评估**方法。

---

## 8. 一次任务的数据流(示例)

以"支付成功但订单未完成"为例:

```
1. Global Lead 接收问题,判断可能涉及 payment + order,也可能涉及 fulfillment。
2. Lead fork 当前任务 run state。
3. Lead 从注册表调用 Payment / Order 业务主 agent,各自 fork 其域 base。
   - Payment Agent:该 payment_id 在支付域是否业务成功?
   - Order Agent:订单完成需要哪些条件?当前为何未完成?
4. Lead 调用 Log / Config 技术域 agent 取证:
   - 回调是否收到、事件是否发出、trace 是否完整。
   - 是否有灰度 / 开关 / 配置变更影响链路。
5. 各 agent 只回传 claim + evidence_refs(窄接口)。
6. Lead 综合:支付成功 → 订单未消费支付事件 → 日志显示回调收到但消费失败
   → 配置显示某灰度开关影响消费链路。
7. Lead 输出最终结论、证据链、风险与下一步动作。
8. 任务产出经沉淀治理,生成相关 base 的 patch proposal。
```

---

## 9. 边界(不做什么)

- 不重写执行核:agent loop / 工具调用 / 文件操作交给 Claude Agent SDK。
- 不把价值押在 agent 数量或调用编排上——那是 commodity。
- 不让 LLM 直接改知识库状态——只能 propose / critique。
- 本系统面向多业务域协作场景,不做个人 / 单用户方向。
